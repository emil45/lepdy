   - In the wrong-tap path, call `onAnswer(false)` is NOT called on wrong tap. Wrong taps give hints and try-again feedback ŌĆö the child retries the same puzzle. `onAnswer(false)` should only be called if there is a "give up" or forced-advance path. For this phase, wrong taps remain retry-in-place (matching existing behavior). Do NOT call `onAnswer` on wrong tap.

   WAIT ŌĆö re-reading the research and existing code: the existing MovementPuzzle does NOT advance on wrong tap. The child retries until correct. So `onAnswer(false)` is never called from MovementPuzzle. The `recordWrong` call that exists in the current code is called on every wrong tap. Since `usePuzzleSession.onAnswer` is meant to advance the puzzle, we should NOT call it on wrong tap. Instead, for wrong-tap progress recording, `usePuzzleSession` needs to handle this differently.

   CORRECTION: Looking at this more carefully ŌĆö `recordCorrect`/`recordWrong` (difficulty tier tracking) was moved INTO the puzzle components in Phase 15-02. The RESEARCH says "Calling recordCorrect/recordWrong inside MovementPuzzle/CapturePuzzle: These calls move to usePuzzleSession.onAnswer()". But `onAnswer` only fires on puzzle completion (correct tap). For wrong taps that don't advance the puzzle, `recordWrong` still needs to be called somewhere.

   RESOLUTION: Keep `usePuzzleProgress` import in both puzzle components ONLY for `recordWrong` on wrong taps. Remove `recordCorrect` (that moves to the session hook's `onAnswer`). The session hook's `onAnswer(true)` calls `recordCorrect`. Wrong taps call `recordWrong` directly in the component since the puzzle doesn't advance.

   Actually ŌĆö simpler approach matching the research anti-pattern guidance: Change `onAnswer` callback signature to be called ONLY on correct answer (puzzle solved). Add a separate `onWrongTap?: (pieceId: ChessPieceId) => void` prop that puzzle components call on wrong taps for difficulty tracking. BUT this overcomplicates the interface.

   FINAL DECISION: The simplest correct approach:
   - Remove `usePuzzleProgress` import from both puzzle components entirely
   - `onAnswer(true)` is called after correct tap (after animation delay) ŌĆö session hook handles `recordCorrect`
   - `onAnswer(false)` is called on wrong tap ŌĆö session hook handles `recordWrong`. The wrong tap does NOT advance the puzzle. The session hook's `onAnswer(false)` calls `recordWrong` but does NOT increment `headIndex`. Only `onAnswer(true)` advances to next puzzle.

   This means `usePuzzleSession.onAnswer` must be updated: `onAnswer(true)` advances headIndex + records correct; `onAnswer(false)` only records wrong + resets streak. THIS IS THE CORRECT DESIGN ŌĆö update the Plan 01 hook description accordingly (the executor should implement this).
